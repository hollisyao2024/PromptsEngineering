# 测试风险识别与缓解矩阵模板

> **目的**：主动识别测试活动中的潜在风险，量化评估风险等级，制定缓解措施，确保测试活动顺利进行与质量目标达成。

**维护者**：QA 专家
**最后更新**：2025-11-06
**版本**：v1.0

---

## 风险评估模型

### 风险等级计算公式
```
风险等级 = 风险概率 × 影响程度
```

### 评分维度说明

| 维度 | 分值范围 | 说明 |
|------|---------|------|
| **风险概率** | 1-5 | 1=极低（< 10%）、2=低（10-30%）、3=中（30-50%）、4=高（50-70%）、5=极高（> 70%） |
| **影响程度** | 1-5 | 1=可忽略、2=轻微、3=中等、4=严重、5=灾难性 |

### 风险等级映射规则

| 风险等级分数 | 风险等级 | 颜色标识 | 响应策略 |
|------------|---------|---------|---------|
| 20-25 分 | **R1（极高风险）** | 🔴 红色 | 立即启动应急预案，高层介入 |
| 12-19 分 | **R2（高风险）** | 🟠 橙色 | 制定详细缓解计划，每日监控 |
| 6-11 分 | **R3（中风险）** | 🟡 黄色 | 列入风险清单，每周回顾 |
| 1-5 分 | **R4（低风险）** | 🟢 绿色 | 记录备案，定期检查 |

## 数据来源与阶段对齐
为确保风险矩阵与 PRD/ARCH/TASK 保持一致，QA 填写时请遵循：
- `风险描述` 与 `影响模块`：来源于 `/docs/PRD.md`（或 `/docs/prd-modules/{domain}/PRD.md`）中识别的 Story/AC，结合 `/docs/ARCH.md`（或 `/docs/arch-modules/{domain}/ARCH.md`）的组件与接口契约定义影响范围与潜在风险。  
- `概率` 与 `影响`：参考 `/docs/TASK.md`（或 `/docs/task-modules/{domain}/TASK.md`）的里程碑、Owner、依赖状态，对任务延迟、环境准备与资源瓶颈的概率/影响进行估算。  
- `缓解措施` 与 `应急预案`：与 TDD/ARCH 任务协调所需的工作项、监控策略与回流路径，并在 `/docs/data/traceability-matrix.md` 中记录关联 Story/Test Case 的状态。  
- 风险状态变化必须同步模块缺陷日志与追溯矩阵，保持 QA 风险、缺陷和 Traceability 之间的可追溯性。

## 风险监控与触发器

### 自动化监控工具

```bash
# 1. 风险评分自动计算（基于实时数据）
npm run qa:calc-risk-score

# 2. 风险趋势分析（对比上周）
npm run qa:risk-trend

# 3. 风险触发器检查（是否需要升级）
npm run qa:check-risk-triggers

# 4. 生成风险报告（Markdown + 热力图）
npm run qa:generate-risk-report
```

```bash
    - 建议: 保持 R4，继续监控
```

## 应急预案

### 场景 1: 测试环境完全不可用（TR-ENV-001）

**触发条件**：测试环境宕机且 2 小时内无法恢复

**应急流程**：
1. **立即行动（0-15 分钟）**：
   - QA Lead 通知所有团队成员暂停测试
   - DevOps 评估恢复时间（< 2 小时 or > 2 小时）
   - 若 > 2 小时，启动备用环境切换流程

2. **切换备用环境（15-45 分钟）**：
   - DevOps 启动 AWS 云备用环境
   - QA 验证备用环境可用性（冒烟测试）
   - 更新 CI/CD 配置指向备用环境

3. **恢复测试（45 分钟后）**：
   - 优先执行 TP0 测试用例（20 个，170 分钟）
   - 并行手动测试关键路径
   - 记录环境故障根因，事后复盘

**回退方案**：若备用环境也不可用，转为纯手动测试 + 发布后补测

---

### 场景 2: 测试时间严重不足（TR-SCHED-001）

**触发条件**：距离发布仅剩 2 小时，测试完成度 < 50%

**应急流程**：
1. **立即决策（0-10 分钟）**：
   - QA Lead + PM + Tech Lead 紧急会议
   - 评估是否推迟发布 or 降低测试覆盖率
   - 决策：若 P0 缺陷 = 0，可降级发布（有风险）

2. **快速测试（10-120 分钟）**：
   - 仅执行 TP0 测试用例（20 个，170 分钟）
   - 若时间不足，执行 Top 10 测试用例（50 分钟，覆盖率 80%）
   - 并行手动测试关键用户路径

3. **发布决策（120 分钟后）**：
   - 若 P0 缺陷 > 0，必须延期发布
   - 若 P0 缺陷 = 0 且 P1 缺陷 < 3，可灰度发布（10% 流量）
   - 在灰度期间继续补测 TP1/TP2

**回退方案**：若灰度期间发现严重问题，立即回滚 + 补测 + 重新发布

---

### 场景 3: 自动化测试大面积失败（TR-TECH-001/002）

**触发条件**：CI 中 > 50% 的自动化测试失败

**应急流程**：
1. **快速分类（0-15 分钟）**：
   - 分析失败原因：真实缺陷 or Flaky Tests or 环境问题
   - 若为环境问题，转 Scenario 1（环境应急预案）
   - 若为 Flaky Tests，暂时禁用 Flaky Tests

2. **手动补测（15-120 分钟）**：
   - 根据测试优先级矩阵，手动执行 TP0 测试用例
   - 记录自动化测试失败原因
   - 评估是否推迟发布

3. **修复自动化（发布后）**：
   - 修复 Flaky Tests 根因（异步等待、数据依赖）
   - 更新自动化测试最佳实践文档
   - 重新启用修复后的自动化测试

**回退方案**：若手动测试也无法覆盖，必须延期发布

## 风险回顾与持续改进

### 风险回顾会议

**频率**：每 2 周一次（与 Sprint Retrospective 合并）

**议程**：
1. **风险状态更新**（15 分钟）：
   - 回顾上次会议的风险项当前状态
   - 识别已缓解的风险（可归档）
   - 识别升级的风险（需加强关注）

2. **新风险识别**（15 分钟）：
   - 团队成员提出新发现的风险
   - 使用风险评估模型打分
   - 决定是否加入风险清单

3. **缓解措施回顾**（20 分钟）：
   - 检查缓解措施的执行进度
   - 评估缓解措施的有效性
   - 调整缓解措施（若效果不佳）

4. **应急预案演练**（10 分钟）：
   - 每次选择 1 个高风险项进行应急预案演练（桌面推演）
   - 记录演练中发现的问题
   - 更新应急预案

### 风险趋势分析

每季度生成风险趋势报告，回答以下问题：
- 哪些风险类型最常见？（环境 > 时间 > 技术 > 数据 > 资源 > 需求）
- 哪些缓解措施最有效？（容器化环境 > 自动化测试 > 测试优先级矩阵）
- 哪些风险被低估？（需求变更、自动化测试稳定性）
- 下季度重点改进方向？

## 常见问题

### Q1: 如何区分"风险"和"问题"？

**A**:
- **风险（Risk）**：可能发生的潜在问题（未来时）
- **问题（Issue）**：已经发生的实际问题（现在时）

示例：
- 风险："测试环境可能宕机" → 记录在风险矩阵
- 问题："测试环境正在宕机" → 立即启动应急预案

### Q2: 风险评分是否需要多人投票？

**A**: 建议采用 Delphi 方法（专家匿名评分）：
1. QA Lead、测试人员、DevOps、开发代表各自评分
2. 取中位数或加权平均值
3. 若差异过大（标准差 > 1），讨论达成共识

### Q3: 风险矩阵需要多久更新一次？

**A**:
- **实时更新**：风险发生、缓解措施完成时同步更新
- **每周回顾**：检查风险触发器，更新风险概率/影响
- **每 2 周回顾**：风险回顾会议，全面评估所有风险
- **每季度审计**：归档已缓解风险，清理过时数据

### Q4: 如何量化缓解措施的有效性？

**A**: 使用以下指标：
- **风险等级下降**：R2 → R3（降低 1 级）
- **实际损失减少**：环境宕机次数从 10 次/月降为 2 次/月
- **恢复时间缩短**：故障恢复时间从 2 小时降为 30 分钟
- **成本节约**：避免的测试延期成本（估算）

### Q5: 如何处理"接受风险"的情况？

**A**: 当缓解成本 > 风险损失时，可接受风险，但需满足：
1. 明确记录接受风险的决策（负责人、日期、理由）
2. 制定监控机制（若风险升级需重新评估）
3. 准备应急预案（若风险真实发生如何应对）
4. 定期回顾（至少每季度重新评估）

示例：
```markdown
### 接受的风险
| 风险 ID | 风险描述 | 风险等级 | 接受理由 | 接受人 | 接受日期 | 下次评估 |
|---------|---------|---------|---------|--------|---------|---------|
| TR-TECH-005 | 跨浏览器兼容性测试不充分 | R3 (6) | 主流浏览器覆盖率 95%，Edge 用户占比 < 2% | @qa-lead | 2025-11-06 | 2026-02-01 |
```
